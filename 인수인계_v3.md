### 3-3. Fable 검수·수정 3건

**① `arena/js/net/net.js` `resolveMode()` — 모드 판정 순서 결함**

수정 전: `auto` 모드가 supabase 키 존재 여부를 wsrelay 단서보다 먼저 검사했다. 수정 후(23~30번째 줄
부근, 함수 내부):
```
// ★ 우선순위 계약(config.js §주석): wsrelay 단서 → supabase → solo.
// Supabase 키는 항상 설정돼 있으므로 supabase를 먼저 검사하면 교내망 릴레이가
// 페이지를 서빙 중이어도 supabase로 붙어 수업 트래픽이 무료 티어를 초과한다. (Fable 검수 수정)
if (hasWsHint) return 'wsrelay';
if (CFG.SUPABASE_URL && CFG.SUPABASE_ANON_KEY) return 'supabase';
```
근거: `config.js` 주석 계약이 wsrelay 단서(WSRELAY_URL 존재 또는 `location.port === '8080'`) →
supabase → solo 순서를 명시. Supabase 키가 항상 설정돼 있어 수정 전 로직대로면 수업 시간에 교내망
릴레이 대신 supabase로 연결되어 무료 티어(월 4매치 규모, v2 §2 트래픽 산정 근거)를 초과할 위험이
있었다. **fdab018에 커밋됨.**

**② `arena/js/main.js` `startMatch()` — 원격 플레이어 팀 병합 결함**

수정 전: presence/join 이벤트로 들어오는 profile은 입장 시점 것(`team:null`)이라 `'peer'` 이벤트만으로는
원격 플레이어의 팀색을 결정할 수 없었다. 수정 후(185~193번째 줄 부근):
```
// ★ 팀 정보 보강(Fable 검수 수정): presence/join의 profile은 입장 시점 것(team:null)이라
//   'peer' 이벤트만으로는 팀색을 알 수 없다. 팀 확정치는 msg 'team'을 병합해 온
//   RoomSession 로스터가 원본이므로, 매치 시작 시점 로스터로 id→team 맵을 만들어 병합한다.
const teamById = new Map((roster || []).map((p) => [p.id, p.profile && p.profile.team]));
adapter.on('peer', ({ id, profile: pf, state }) => {
  ...
  if (!merged.team && teamById.has(id)) merged.team = teamById.get(id);
```
미수정 시 원격 플레이어 전원이 주황(OX 팀색)으로 렌더되는 결함. **fdab018에 커밋됨.**

**③ `arena/js/core/tps.js` — 카메라 오클루전 추가 (미커밋, §1 참조)**

사용자가 스모크 테스트(§0-7)에서 직접 발견한 "벽면에 카메라가 파묻혀 시야가 가려지는" 문제의 수정.
- 상수(19~20번째 줄): `CAM_COLLIDE_MARGIN = 0.3`(벽 표면에서 띄울 여유, m), `CAM_MIN_DIST = 0.6`
  (카메라가 피벗에 붙을 수 있는 최소 거리, m)
- 모듈 함수 `raySlabAABB(ox,oy,oz,dx,dy,dz,box,m,L)`(24번째 줄~): 선분 vs 마진 확장 AABB 슬랩 교차
  판정, 첫 교차 지점의 t값 반환. THREE.Raycaster 없이 순수 산술 — 프레임당 신규 할당 0
- 인스턴스 메서드 `_occludeToward(target)`(234번째 줄~): pivot→target 선분이 콜라이더와 교차하면
  target을 첫 교차 지점 앞(마진만큼)으로 당김
- `_updateCamera(dt)`·`_syncCameraInstant()` 양쪽에서 이상적 위치(`_idealPos`)와 보간 중인 위치
  (`_camPos`) **모두**에 `_occludeToward` 적용 — 목표만 가리면 보간 도중 잠깐 벽을 통과하는 경우가
  남기 때문
- 부수 변경: 두 메서드에서 `this._pivot` 갱신 순서를 카메라 계산보다 앞으로 이동(오클루전 계산이
  최신 피벗을 써야 하므로)

**이 수정은 사용자가 재검증하지 않은 채 세션이 종료됐다. 다음 세션 최우선 작업은 (a) 커밋 여부

## 5. git 추적 상태

- 로컬 `main` = origin/main 해시 일치(`fdab018`), ahead/behind 없음
- 유일한 작업트리 변경: `arena/js/core/tps.js`(미스테이지, §1·§3-3③)
- 스테일 `.git/index.lock` 문제는 §0-2에서 사용자 cmd로 해소됨(v2 §5.2 해소 — 이번 세션 재발 없음).
  단, 샌드박스에서 쓰기 git 명령(add/commit/push)은 여전히 실행 불가(권한 문제 상시)이므로 다음
  세션도 커밋·푸시는 사용자 PC cmd에서 수행해야 한다
- 참고: `git status --short`에 한글 경로가 `D`+`??` 쌍으로 뜨는 것은 유니코드 정규화(NFC/NFD) 표시
  차이일 뿐 저장소 손상이 아니다(§1 각주) — 이번 세션에는 나타나지 않았으나 재발 시 실제 삭제와
  구분해서 판단할 것

## 6. 미해결·이월 문제

### [해소됨] v2 §5.1 — 하위 에이전트 A·B 미투입

### [해소됨] v2 §5.2 — arena 전체 미커밋 + index.lock
이번 세션 §0-2·§0-6에서 커밋(`9310c8d`, `fdab018`) + push 완료. **완전 해소.** 단 §3-3③(오클루전
수정)은 그 이후 이뤄진 별도 변경이라 이 해소 범위 밖 — §1·§4 참조.

### [new, core] 샌드박스 마운트 동기화 플래핑
이번 세션 중 bash 도구가 `arena/js/net/net.js`를 5014바이트로 잘린 상태로 보여준 적이 있었으나
호스트 원본(Windows 파일시스템)은 정상이었다. v1 §4-3(작업트리 실제 파일 잘림 사고)과는 다른
층위의 문제 — 이번엔 **표시(읽기 스냅샷) 문제**이지 파일 손상이 아니었다. 구분 방법: bash에서 파일이
잘려 보이면 즉시 Read 도구(호스트 경로)로 재확인할 것. 실제 손상인지 표시 잡음인지는 크기 비교로
가른다.

### [new, minor] 회전된 엄폐 블록의 시각 vs 충돌 불일치
`arena/js/world/arena.js`의 콜라이더가 회전을 무시하는 AABB라서, 맵에 회전된 엄폐물이 있으면 시각적
모서리와 실제 충돌 경계가 어긋난다. Phase A 범위에서는 방치 가능하나 Phase B 전투 판정 정밀도에
영향을 줄 수 있어 기록.

### [new, minor] wsrelay status count 미갱신
`arena/js/net/wsrelay.js`의 status 인원수가 최초 입장(peers) 시점에만 갱신되고, 이후 join/leave
이벤트로는 갱신되지 않는다.

### [new, minor] RemotePlayers 팀 변경 시 아바타 색 재생성 안 됨
`arena/js/world/players.js`가 원격 플레이어를 생성한 뒤 팀이 바뀌면(늦은 입장자가 매치 중간에 팀
배정을 받는 등) 아바타 색을 다시 만들지 않는다. 매치 중 늦은 입장자에 한정된 엣지 케이스.

### [new, minor] supabase presence 경합
`setTimeout(handlePresenceSync, 0)` 형태의 방어 코드는 존재하나(에이전트 B 보고), 실기기 다중 접속
환경에서 검증된 적이 없다. §4 TODO의 멀티 실검증 항목과 함께 확인 필요.
